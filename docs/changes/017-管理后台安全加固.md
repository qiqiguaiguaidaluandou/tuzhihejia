# 017 管理后台安全加固：破坏性操作二次确认 + 末位管理员保护

- 状态：已完成
- 日期：2026-06-30
- 关联提交：（待回填）

## 需求 / 改动

对 Blazor 管理后台（`TZHJ.Gateway`，[012](012-Blazor管理后台.md)）做安全向优化，聚焦两处"点错即出事、且现状无防护"的风险点：

1. **破坏性操作无二次确认**。删除角色（会**级联**剥夺关联用户的数据范围，见 `TzhjDbContext` 上 `UserRole` 的 `OnDelete(Cascade)`）、停用用户，都是点一下立即落库，没有任何确认，误点代价大。
2. **可能停用最后一个管理员**。`SetActiveAsync` 原本只防"停用自己"；但另一个管理员完全可以把**最后一个启用的管理员**停掉，导致**无人能再进后台**（管理员标记只在建用户时设，无降级入口，唯一收紧路径就是停用）。

> 评估后**未做**的项：角色编辑乐观并发（管理员人数极少、真实冲突概率低，且需改 `TZHJ.Core` 契约层，性价比低）；"数据范围去重"已在 `AdminRoles.AddRange` 添加时拦截，无需再改。

## 方案

- **A 二次确认**：新增**通用**模态组件 `ConfirmDialog`，纯 CSS（复用现有 `.card/.btn` 风格 + 新增遮罩样式），InteractiveServer 下用 `TaskCompletionSource<bool>` 把"点确认/取消"转成可 `await` 的结果，不引入 JS interop。删角色弹窗中把"关联 N 人"亮出来（N 取现成的 `RoleSummary.UserCount`）。
- **B 末位管理员保护**：在**服务端** `SetActiveAsync` 加判据——服务端兜底，绕过 UI 走 REST `/api/admin/*` 也拦得住，比仅前端禁用可靠。

## 实现

1. `src/TZHJ.Gateway/Components/Shared/ConfirmDialog.razor`（新增）：`ShowAsync(title, body, confirmText, danger)` 返回 `Task<bool>`；点遮罩/取消=false，确认=true；重复弹出时把上一个悬挂的 `await` 当取消收尾。
2. `src/TZHJ.Gateway/Components/App.razor`：新增 `.modal-overlay/.modal-card/.modal-body/.modal-actions` 样式（内联样式表，沿用无外部依赖的约定）。
3. `src/TZHJ.Gateway/Components/_Imports.razor`：补 `@using TZHJ.Gateway.Components.Shared`，使 `<ConfirmDialog>` 可解析。
4. `src/TZHJ.Gateway/Components/Pages/AdminRoles.razor`：放 `<ConfirmDialog @ref="_confirm" />`；删除按钮改传整行 `RoleSummary`；`Delete` 先弹确认（关联用户>0 时提示"关联 N 个用户，删除后将失去这部分数据范围"），取消则不执行。
5. `src/TZHJ.Gateway/Components/Pages/AdminUsers.razor`：放 `<ConfirmDialog @ref="_confirm" />`；`SetActive` 在**停用**分支先弹确认（"停用后无法登录，账号与审计保留，可随时重启用"），启用不弹。
6. `src/TZHJ.Gateway/Auth/AdminService.cs` `SetActiveAsync`：在"启用中的管理员被停用"时，统计其他启用管理员数，为 0 则 `Fail("系统至少需保留一个启用的管理员，不能停用。")`。仅在 `!isActive && user.IsActive && user.IsAdmin` 时校验，避免对无变化的重复停用误拦。
7. `tests/TZHJ.Tests/Auth/AdminServiceTests.cs`：新增 `Cannot_deactivate_last_active_admin`（由他人发起仍被拦）、`Can_deactivate_admin_when_another_active_admin_exists`（有备用管理员时放行）。

不变：管理后台其余页面/逻辑、权限模型（数据范围 only）、Cookie 鉴权、登录与日志、整体风格。

## 验证

`dotnet build src/TZHJ.Gateway`（`-p:EnableWindowsTargeting=true`）0 错；`dotnet test tests/TZHJ.Tests -p:EnableWindowsTargeting=true` **61/61** 通过（原 59 + 新 2）。

确认弹窗为浏览器交互，沙箱无 GUI 待真机验证：角色管理点"删除"→ 弹窗显示角色名与关联人数 →"取消"不删 /"删除"才落库；用户管理点"停用"→ 弹窗确认；停用唯一启用管理员（含经 REST）→ 服务端拒绝。

## 备注 / 待办
