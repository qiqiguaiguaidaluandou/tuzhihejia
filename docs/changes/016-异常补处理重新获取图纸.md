# 016 异常补处理：重新获取图纸（PLM 补图后回拉）

- 状态：已完成
- 日期：2026-06-29
- 关联提交：e04ebbb

## 需求 / 改动

取数时刻 PLM 里可能还没传该料号图纸，操作员在挑图/核价里看不到图，于是把这行**挂异常**（原因如"没有图纸"）。之后 PLM 把图纸补上了，但系统没有重新取图的入口——操作员无法把新图纸拿到本地批次文件夹里查看、继而结案。

希望在**补处理页**加一个「重新获取图纸」按钮（放在「重新回传」按钮**前面/左侧**），点击后：

1. 去 PLM **重新拉取该行（料号）的图纸**；
2. 把拉到的图纸落到**该批次对应的服务端文件夹**，并**同步到本地**批次文件夹；
3. PLM **真补上了** → 提示"图纸获取成功"；**仍没补** → 提示"重新获取图纸失败"。

「重新获取图纸」只负责把图纸搞到位，**异常仍开着**；操作员看完图再点「重新回传」结案（沿用 [003](003-异常补处理单行回传.md) 的单行回传）。

## 方案

整体：**新增一个服务端"按行重新取图"端点 + 客户端补处理页一个命令**，复用现成的 PLM 取图、落盘、同步、下载设施，不碰定时采集主链路。

链路：补处理页点「重新获取图纸」→ `POST /api/batch/refetch-drawing` →（服务端）复用 `PlmClient.FetchDrawingsAsync` 拉该料号图纸 → 取不到回 `Found=false`（客户端提示失败）/ 取到则写进该批次服务端文件夹并回 `Found=true,Files` →（客户端）按 `Files` 经 `/api/sync/download` 下载到本地批次目录、并入该行 manifest 图纸清单 → 提示"图纸获取成功"。

关键决策与依据：

- **不新增取图接口，直接复用 `PlmClient.FetchDrawingsAsync`**。该接口本就接收料号集合（支持批量/单个），传 `new[]{料号}` 即可。端点直接注入 `PlmClient`（`Program.cs` 已 `AddHttpClient<PlmClient>()`）调用——直连走的是**真实 PLM** 路径，不受 EBS 取数是否走 Fake 影响，"重新获取图纸"本就该去问真实 PLM。
- **服务端只写文件、不写 manifest**。服务端目录无 `_manifest.json`，同步清单 `/api/sync/catalog` 的文件列表来自目录列举（`ListFilesAsync`）。所以服务端只要把图纸写进已存在的批次目录，sync 即能列到；为即时反馈，客户端不等周期同步，拿到 `Files` 后立即逐个下载到本地。
- **客户端用"只更新行图纸清单"的精准方法，不用 `SaveBatchAsync`**。补处理页加载时已对该行调过 `Row.Restore()`（内存中把状态由 `Exception` 改成 `Pending/Done`）；若此时调 `SaveBatchAsync` 会把"已解除异常"的状态**提前**写进本地 manifest（尚未回传），造成本地批次该行状态与异常池不一致。故新增 `AddRowDrawingsAsync`：只把新图纸文件名并入该行 manifest 的 `Drawings`（去重），**不动行状态/字段值**。
- **常显、两个流程都加**。挑图与核价取数都会从 PLM 富化图纸、都可能因缺图挂异常，故两流程补处理页都提供该按钮。可见性以"来源批次/行已加载"为准（`CanRefetch`），**与回传权限解耦**——没有回传权限也能补图（补图是取，不是回传）。
- **异常不自动结案**。重新取图成功只把图纸落到位并提示，异常仍留在池中；与 [003](003-异常补处理单行回传.md) 的"看完图→重新回传→结案"衔接。

## 实现

1. `src/TZHJ.Core/Contracts/Http/SyncDtos.cs`：新增 `RefetchDrawingRequest`（Flow/BatchId/GroupName/RowKey/MaterialCode）与 `RefetchDrawingResult`（`Found` / `Files` / `Message`）。
2. `src/TZHJ.Gateway/Stores/IServerBatchStore.cs` + `FileServerBatchStore.cs`：新增 `AppendDrawingsAsync`——向已存在批次目录追加图纸文件，复用同批次写锁（`LockFor`），不重写 Excel。
3. `src/TZHJ.Gateway/Endpoints/ApiEndpoints.cs`：新增 `POST /api/batch/refetch-drawing`。校权限 → 查 `BatchRegistry`（无则 404）→ 注入 `PlmClient` 调 `FetchDrawingsAsync(new[]{料号})` → 无图回 `Found=false`（不改批次、不留成功痕迹）→ 有图则 `AppendDrawingsAsync` 落盘 + 刷新 `LastModified` + 记 `RefetchDrawing` 操作日志 → 回 `Found=true,Files`。
4. `src/TZHJ.Core/Contracts/IDataGateway.cs` + `src/TZHJ.Infrastructure/Gateways/Http/HttpDataGateway.cs`：新增 `RefetchDrawingAsync`（POST 上述端点，反序列化 `RefetchDrawingResult`）。
5. `src/TZHJ.Core/Contracts/ILocalBatchStore.cs` + `src/TZHJ.Infrastructure/Storage/LocalBatchStore.cs`：新增 `AddRowDrawingsAsync`——读 manifest、定位该行、去重并入图纸文件名、写回（无 manifest 则跳过，扫盘前缀仍可识别）；不改行状态。
6. `src/TZHJ.App/ViewModels/ExceptionResolveViewModel.cs`：新增 `CanRefetch`（`[NotifyCanExecuteChangedFor]`）与 `RefetchDrawingCommand`。`LoadAsync` 在该行就位后置 `CanRefetch=true`（不要求回传权限）。命令：调 `RefetchDrawingAsync` → `Found=false` 时 `_dialog.Error("重新获取图纸失败：PLM 中暂无…")`；`Found=true` 时逐个 `DownloadFileAsync`+`WriteSyncFileAsync` 到本地（来源批次在 `Done`）、同步内存 `Row.Model.Drawings`（去重）→ `AddRowDrawingsAsync` 持久化关联 → 记操作日志 → `_dialog.Success("图纸获取成功：已拉取 N 张…")`。
7. `src/TZHJ.App/Views/ExceptionResolveView.xaml`：把信息条右侧的「打开来源批次」「重新回传」收进一个横向 `StackPanel`，顺序 `[打开来源批次（含图纸）] [重新获取图纸] [重新回传]`——「重新获取图纸」位于「重新回传」前面（左侧）；用默认（次要）样式与主操作"重新回传"区分，图标 `ArrowClockwise`，可见性绑 `CanRefetch`。
8. `tests/TZHJ.Tests/Sync/Fakes.cs`：`FakeLocalBatchStore` 补 `AddRowDrawingsAsync`、`FakeDataGateway` 补 `RefetchDrawingAsync`（均为最小桩，返回默认）。

不变：挂起异常的产生逻辑、整批/单行回传、定时采集（`DataIngestionService`）、`PlmClient` 取图实现、权限模型、整体风格。

## 验证

`dotnet build`（Gateway / Core / Infrastructure，及 App 加 `-p:EnableWindowsTargeting=true`）0 错；`dotnet test tests/TZHJ.Tests -p:EnableWindowsTargeting=true` **59/59** 通过。

WPF 真机待验：补处理页出现「重新获取图纸」（在「重新回传」左侧）→ PLM 已补图时点击 → 提示"图纸获取成功"、点「打开来源批次（含图纸）」可见新图、再「重新回传」结案；PLM 仍无图时点击 → 提示"重新获取图纸失败"。

## 备注 / 待办

- 真机需 `Ebs:Enabled` + `Plm:DrawingUrl` 实配后验证真实回拉；Fake/未配 `Plm:DrawingUrl` 时 `FetchDrawingsAsync` 返回空，"重新获取图纸"恒提示失败（符合"无真实 PLM 即无图"语义）。
- 文件名按 PLM 返回名（`SafeFileName` 后）落盘，与首次取数一致；跨料号同名碰撞为既有取数链路同等的小概率边界，本期不额外处理。
