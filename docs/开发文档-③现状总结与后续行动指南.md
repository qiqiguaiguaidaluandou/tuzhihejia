## 2. 核心架构深度解析

### 2.1 “本地即状态”设计
系统不依赖本地数据库存储作业进度。
*   **真相源**：用户的本地文件夹（核价/挑图根目录）。
*   **映射逻辑**：一个子文件夹 = 一个批次；文件夹下的 `.xlsx` 文件 = 该批次的业务数据与状态快照。
*   **优势**：极强的离线能力，支持操作员将文件夹拷贝到另一台安装了客户端的机器上无缝继续作业。

### 2.2 字段配置化驱动
*   **机制**：作业网格（DataGrid）的列由 `FieldDefinition` 列表驱动。
*   **下发流程**：客户端启动登录后，从后端 `/config` 端点获取最新的 `ClientConfig`，动态生成 UI 界面。
*   **业务价值**：增加、删除字段或更改必填项，仅需修改后端配置，无需客户端重新发版。

## 3. 后续详细开发任务清单

### 第一阶段：内功加固（不依赖外部接口）

#### 3.1 [已完成] 提交闸门 (Submit Gate)
*   **现状**：已通过 `BatchWorkViewModel.CanSubmit` 属性与 `PendingCount` 联动实现。
*   **效果**：只要网格中存在“待处理”行，回传按钮自动禁用，强制要求操作员对所有行完成“填值”或“挂起异常”。

#### 3.2 图纸完整性校验 (UI 辅助)
*   **目标**：消除“有记录无文件”导致的作业困惑。
*   **逻辑**：在 `LocalBatchStore.CheckIntegrity` 中完善附件检查。
*   **注意**：由于回传报文不含图纸二进制流，此校验仅作为 UI 警告提示（图标显示），**不阻断**回传逻辑。

#### 3.3 [已完成] 本地扫描与异常同步
*   **现状**：已在 `BatchListViewModel` 中通过 `FileSystemWatcher` 实现文件夹增删改的去抖自动刷新。
*   **异常闭环**：异常池入池（提交时）、异常展示、异常解决（补回传+自动移出）已在 `Exception*ViewModel` 中完整实现。

### 第二阶段：数据持久化（后端落地） —— **当前核心任务**

#### 3.4 [已完成] PostgreSQL 审计存储
*   **实现内容**：已完成 `TzhjDbContext` 和 `PgAuditStore` 的编写。
*   **技术栈**：EF Core + Npgsql (PostgreSQL)。
*   **状态**：已生成初始迁移（InitialCreate），待配置 `appsettings.json` 连接字符串并执行 `database update` 即可正式启用。

#### 3.5 配置中心持久化
*   **目标**：按人/按机精细化配置。
*   **任务**：将目前硬编码在 `InMemoryConfigStore` 中的字段定义和时间窗逻辑搬入数据库，支持管理员通过数据库快速调整业务规则。

### 第三阶段：外部对接（待规格对齐后启动）

#### 3.6 防腐层 (ACL) 真实替换
*   **EBS**：按照工号过滤，拉取待处理物料及图纸元数据。
*   **PLM**：实现图纸附件下载流，支持版本校验，确保取到的是建批次时的有效版。
*   **SRM/EBS**：实现回传幂等性校验，确保网络波动导致的多发请求不会造成业务逻辑错误。

## 4. 验证与质量保障建议
1.  **真机验收 (D1)**：必须在 Windows 10/11 真实环境下，针对不同 DPI 缩放（如笔记本常见的 125%/150%）验证 WPF 布局的鲁棒性。
2.  **补拉压测**：模拟大规模补拉（一次拉取 50+ 批次），验证异步下载和文件系统的并发稳定性。
3.  **错误翻译**：通过 `FriendlyError.cs` 将“404 Not Found”或“Socket Timeout”翻译为“服务器暂时无法连接，请稍后重试或联系 IT”。
